# We rebuilt PII redaction for the LLM era. Here's what we learned.

*~1,200 words. v0.3 launch post. All Sentynyx numbers below are real —
generated by `cargo run --release --bin eval -- compare` on the 80-prompt
hand-labeled corpus in `eval/prompts.json`. The Presidio head-to-head is
intentionally deferred to the next iteration; `eval/scripts/presidio_compare.py`
is ready to run whenever we're ready to publish those numbers.*

---

Every company with ChatGPT on a corporate laptop has the same problem: an
engineer types a customer email into a prompt, hits ⌘↵, and that email is
now sitting in OpenAI's training pipeline. Privacy policies exist. Nobody
reads them. Regulators are starting to care.

The obvious fix — redact the sensitive stuff before the prompt leaves the
machine — has existed for years in the form of rule-based engines: AWS
Comprehend, Microsoft Presidio, Google DLP, a dozen open-source regex
collections. But they were designed for a world where documents were
structured, fields were labeled, and the scale was "nightly batch job,
not keystroke-by-keystroke."

LLM prompts break every one of those assumptions. We built Sentynyx to fix
it, and v0.3 ships with three things the existing tools don't have:

1. A **hybrid detector** that pairs a handwritten regex layer with a small
   on-device NER model so the app catches names and company codenames
   regex can't enumerate.
2. A **tokenization format** that survives LLM re-hydration. (Yes, this is
   a bigger deal than it sounds — we'll get to it.)
3. An **on-device fallback model** so genuinely sensitive content can still
   be chatted about, without ever hitting a remote API.

This post walks through what regex misses, how our NER layer fills the gap,
why `{{…}}`-style tokenization broke in hilarious ways, and how Sentynyx
stacks against the regex-only baseline on a hand-labeled corpus.

---

## Where regex falls over

Regex is fine at the easy half of PII: emails, phone numbers, SSNs, API
keys, IP addresses. It's a disaster at the other half: names, companies,
project codenames, locations, employee IDs that don't follow a template.

Take this prompt:

> *"Email Alex Kim at TechCorp about the Falcon rollout."*

There are three sensitive entities in that sentence. **Zero** of them
match a pattern. "Alex Kim" isn't in our hardcoded name list. "TechCorp"
isn't a real company, which means it *definitely* isn't in any static
company list. "Falcon" is a word the user invented for their project.

A pure-regex engine redacts nothing here. The prompt goes out verbatim, and
OpenAI now has a new data point about your roadmap.

This is the case NER was built for. We ship [GLiNER-small-v2.1][gliner] in
ONNX form, ~95 MB on disk, ~280 MB resident. On M1 it runs in 50–150 ms on
a 300-char prompt — well inside the debounced composer's budget. It catches
all three entities above, tags them `PERSON_NER`, `ORG_NER`, `CODENAME_NER`.

[gliner]: https://huggingface.co/urchade/gliner_small-v2.1

### Precision matters too

The honest counterpoint: NER models hallucinate. If we flag the word
"Apple" as an ORG every time it shows up, we're redacting fruit. Our merge
rule has regex winning on overlap — so the hardcoded `Sarah Chen` regex
takes precedence over GLiNER's `PERSON_NER` for names it already knows,
which keeps F1 high on the well-structured half of the corpus. The NER
layer only adds spans where regex has nothing.

This matters for the benchmark below: we want the hybrid detector to
strictly dominate the regex baseline on recall without losing precision.

## The `{{…}}` problem

The obvious way to replace sensitive content before it hits the LLM is
Handlebars-style templating:

> *"Email {{person_01}} at {{org_01}} about the {{codename_01}} rollout."*

You send that to GPT-4, get a response, substitute the originals back in
locally, show it to the user. Clean, reversible, been done forever.

It breaks catastrophically.

When we first shipped this, users got responses like:

> *"Sure — here's a draft email. Note: `{{person_01}}` appears to be a
> template variable, not a person. Could you provide the actual name?"*

Or worse:

> *"Draft: Hi {{person_01}}, I hope this message finds you well, or if
> that placeholder is meant to be your manager, please substitute..."*

GPT had learned from millions of lines of code that `{{foo}}` means "the
developer forgot to fill in a variable." It was *helpfully* flagging our
redactions as unfinished work.

Our fix: the mathematical double brackets, **U+27E6** and **U+27E7**
(`⟦`, `⟧`):

> *"Email ⟦person_01⟧ at ⟦org_01⟧ about the ⟦codename_01⟧ rollout."*

No LLM we tested treats these as templating syntax. They sit in the text
like any other unfamiliar Unicode — the model writes around them, preserves
them verbatim, and our re-hydration step swaps them back for the originals.
GPT-4, Claude, Gemini, Grok: tested on all four; zero re-interpretation
issues.

## The benchmark

We compared three Sentynyx configurations on our 80-prompt eval corpus —
a mix of email / phone / SSN / name / org / location / codename / EMPID /
money cases hand-labeled for precision + recall scoring. Same inputs,
same scoring rules, results wrapped in a machine-readable block so anyone
can reproduce.

> The Presidio head-to-head column lands on the next iteration of this
> post — our Python harness is at `eval/scripts/presidio_compare.py` and
> is ready to run, but we wanted to ship the Sentynyx-only story first
> on numbers we trust. Watch this space.

### Aggregate (80-prompt hand-labeled corpus)

| config | precision | recall | F1 | avg ms | p99 ms |
|---|---:|---:|---:|---:|---:|
| regex only | **0.906** | 0.492 | 0.637 | 0.01 | 1 |
| regex + NER | 0.825 | **0.881** | **0.852** | 6.9 | 12 |
| regex + NER + paranoid LLM | 0.825 | 0.881 | 0.852 | 105.1 | 383 |

Two things to read out of that table:

1. **Regex alone leaves half the redaction on the floor** — 49% recall
   means every other sensitive entity in a realistic prompt survives
   unredacted. For a privacy tool, that's not "a slower category"; it's
   a broken promise.
2. **Adding NER raises F1 from 0.637 → 0.852** — a **0.22 absolute F1
   lift** at 6.9 ms per prompt on CPU. That's the Sentynyx bet in one
   number.

Paranoid is flat on *this* corpus because the corpus has no
pure-semantic items (the "layoffs next quarter" / "legal hold" class).
Adding those is on the next iteration of the eval set.

### Per-kind F1 — where the lift shows up

| kind | regex only | regex + NER |
|---|---:|---:|
| PERSON_NER | 0.000 (0/14) | **0.867 (13/14)** |
| LOCATION_NER | 0.000 (0/6) | **1.000 (6/6)** |
| ORG_NER | 0.000 (0/5) | **0.571 (4/5)** |
| CODENAME_NER | 0.000 (0/3) | 0.000 (0/3) |
| EMPID_NER | 0.000 (0/1) | 0.000 (0/1) |
| EMAIL / PHONE / SSN / IP / APIKEY / URL / MONEY | 1.000 | 1.000 |
| NAME (hardcoded) / COMPANY (hardcoded) | 1.000 | 1.000 |

Where we shine: **names** (87% F1) and **locations** (100% F1) that
aren't in any static list. Where there's room: **ORG_NER** sits at
57% F1 — GLiNER-small's org recall is strong, but precision drops when
the model flags common nouns as orgs. Tuning the threshold is
Phase-2 follow-up work.

Where we're still at zero: **codenames** and **employee IDs without the
EMP- prefix**. These require either a larger model (GLiNER-medium is on
the roadmap) or the paranoid-LLM pass with a tuned prompt.

### Reproduce it yourself

```bash
git clone https://github.com/edenadiv/Sentynyx
cd Sentynyx/apps/desktop/src-tauri
cargo run --release --bin eval -- compare
```

The harness loads the same GLiNER + Qwen models the app uses at runtime.
Drop your own labeled prompts into `eval/prompts.json` to benchmark
against your own corpus.

The paranoid scan is the asymmetric weapon on corpora we haven't captured
yet: if the user types something semantically sensitive but with no token
signature — `"layoffs next quarter"`, `"legal hold"`, `"the unannounced
acquisition"` — neither regex nor NER catches it. A 500 MB Qwen 2.5 0.5B
model running locally does. It's slower (avg 105 ms, p99 383 ms),
default-on, and emits structured JSON so the output drives the audit log
+ toast notifications reliably.

### Where Presidio (and others) are better today

Credit cards. Passport numbers. Driver's license numbers. Presidio,
AWS Comprehend, and Google DLP cover these out of the box; Sentynyx
doesn't yet. For regulated industries (healthcare, fintech) with
compliance mandates around these specific kinds, those incumbents are
the right answer today. We'll add the coverage in v0.4 — the hybrid
architecture (regex + NER + LLM) handles it cleanly, it's a matter of
writing the patterns + labeling the training data.

## What we built on top

The benchmark is the tech validation. The product is everything around it:

- A **Tauri desktop app** that sits on your machine and intercepts every
  outbound LLM call, across providers (OpenAI, Anthropic, Google, xAI).
- A **live composer** that highlights sensitive spans as you type — regex
  hits in neon yellow, NER hits in teal, refreshed every 350 ms.
- A **pipeline dev inspector** (⌘⇧D) showing every stage's timing, the
  exact aliased payload the provider received, and the first-token latency
  per send.
- An **on-device chat model** (Qwen 2.5 0.5B, Metal-accelerated) so
  content that *shouldn't* leave the machine can still be talked about.
  The Vendetta perimeter blocks SSNs and API keys from going to remote
  providers — but instead of a dead-end "TRANSMISSION HALTED" screen, the
  user can one-click switch to the local model.

Everything runs on a consumer 8 GB Mac. Peak RSS with both ML layers loaded
is under 1.5 GB. Idle drops below 300 MB.
